# MES-lite 配置化页面框架规划

本文记录 MES-lite 吸收低代码平台优点的实现方向。

目标不是引入外部低代码平台，而是在当前代码仓库内建立一套适合 MES 业务的配置化页面框架，让常见页面更统一、更快开发、更容易维护。

## 1. 基本判断

MES-lite 的库存、成本、工单、权限、审计等逻辑需要代码完全可控，不适合整体迁移到通用低代码平台。

但低代码平台的以下优点值得吸收：

- 表格、筛选、分页、排序可配置。
- 表单字段和布局可配置。
- 按钮、操作、状态标签统一。
- 权限和页面动作统一绑定。
- 常见 CRUD 页面可以快速生成。
- 页面结构一致，后续统一改版成本低。

## 2. 设计原则

1. 业务逻辑继续手写，不能把库存、成本、状态流转交给配置隐式处理。
2. 页面外壳配置化，减少每个页面重复写工具栏、筛选、表格、卡片、弹窗。
3. 页面注册、字段定义和业务动作必须是 TypeScript 代码，跟随 Git 版本管理；菜单名称、层级、排序和显示状态允许作为独立配置保存。
4. 权限、审计、状态流转必须显式声明，不能隐藏在页面配置里。
5. 桌面端和移动端继续分开验收，同一份资源配置可以渲染不同布局。

## 5. 优先落地顺序

### 第一阶段：统一展示外壳

1. 抽出统一按钮、状态标签、视图切换、分页组件。
2. 抽出统一顶部工具栏和筛选弹窗。
3. 抽出统一列表/卡片容器。
4. 先迁移作业指导书和物料管理两个页面。

### 第二阶段：资源配置化

1. 将字段、筛选、排序、操作按钮整理为资源配置。
2. 将权限资源和页面动作绑定到配置。
3. 将字段显示控制、默认排序、分页大小纳入配置。

### 第三阶段：表单和详情配置化

1. 抽出通用表单网格。
2. 抽出通用详情弹窗。
3. 文件上传、PDF/图片查看器统一接入。
4. 移动端使用同一资源配置渲染底部抽屉和卡片布局。

## 6. 不做的事情

- 不引入外部低代码平台作为运行依赖。
- 不让非技术用户在生产环境随意改字段结构。
- 不用数据库配置生成核心业务流程。
- 不把库存成本、单据状态流转做成纯配置规则。

## 7. 成功标准

- 新增一个常规业务页面时，不再重复写工具栏、筛选、表格、卡片、弹窗基础结构。
- 同类按钮、状态、筛选、分页、字段显示完全一致。
- 修改桌面端或移动端布局规则时，不需要逐页手工改。
- Git 历史能清楚看到资源配置、页面组件和业务逻辑分别变更了什么。

## 8. 可配置菜单开发规范

### 8.1 基本定位

MES-lite 的终端页面和业务逻辑由代码实现，菜单只是这些固定页面的可配置入口。

可配置菜单不等于动态生成数据表、表单、接口或核心业务流程。当前产品定位为“可配置工作台”，不宣称为完整低代码平台。

```txt
代码中的固定页面注册表
        ↓
系统管理员基础菜单配置
        ↓
当前用户权限过滤
        ↓
终端用户个人布局配置
        ↓
最终显示菜单
```

### 8.2 页面注册表

每个可进入的功能页必须先在代码中注册，至少声明：

```ts
{
  pageKey: 'processTemplates',
  defaultName: '加工工艺',
  permissionResource: 'system',
  supportedClients: ['desktop', 'mobile'],
  defaultIcon: 'process',
}
```

纯计算工具同样必须注册为稳定页面，例如 `sawingCost`（锯切成本）；后续菜单配置上线后，管理员可将它放入“工艺管理”、“成本工具”或其他自定义目录。

- `pageKey` 是永久唯一标识，发布后不得因菜单改名而修改。
- 页面路由、权限资源、业务接口和页面组件不得由菜单配置直接改写。
- 新功能只需新增页面注册项，不应手工重排整套菜单代码。
- 管理员未分配新页面时，默认进入“未分类功能”，不覆盖现有菜单。

### 8.3 系统管理员基础配置

系统管理员负责整个组织的菜单结构和安全边界，可以：

- 新增、改名、排序、启用或停用一级目录。
- 修改二级菜单的显示名称和图标。
- 将已注册页面分配到指定一级目录。
- 设置默认排序、默认显示状态和移动端默认常用入口。
- 设置哪些菜单允许用户自行隐藏或调整。
- 选择或恢复系统默认布局。

管理员修改的是显示入口，不得通过改名、隐藏或移动菜单改变页面的真实权限。

### 8.4 终端用户个人配置

终端用户在管理员基础菜单和自身权限范围内，可以：

- 调整一级目录的显示顺序。
- 调整同一目录内二级菜单的顺序。
- 折叠或展开目录。
- 隐藏自己不常用且管理员允许隐藏的入口。
- 设置移动端常用入口。
- 恢复管理员的默认布局。

终端用户不能：

- 修改管理员设定的一级、二级菜单名称。
- 启用管理员停用的页面。
- 看到没有权限的页面。
- 修改 `pageKey`、路由或权限资源。
- 修改其他用户的布局。

### 8.5 个人配置存储

第一阶段使用浏览器 `localStorage` 保存个人布局：

```txt
mes-lite.menu.user-layout.v1
```

示例：

```json
{
  "groupOrder": ["production", "materials", "analysis"],
  "itemOrder": {
    "production": ["orders", "dispatch"],
    "materials": ["materials", "stocks", "materialIn"]
  },
  "collapsedGroups": ["analysis"],
  "hiddenItems": ["return"],
  "mobileFavorites": ["dashboard", "orders", "dispatch", "stocks"]
}
```

- 不默认使用浏览器直接改写任意本地文件。
- 第二阶段将个人布局保存到用户账户，实现跨设备同步。
- 需要提供 JSON 导入、导出，用于备份和迁移个人布局。
- 本地配置必须包含版本号，升级时忽略已失效字段，不阻断页面启动。

### 8.6 菜单合并顺序

最终菜单必须按以下顺序生成：

1. 读取代码中的固定页面注册表。
2. 应用系统管理员的基础菜单配置。
3. 根据当前用户权限删除无权入口。
4. 应用当前用户的排序、折叠、隐藏和移动端收藏。
5. 忽略已下线或不存在的旧 `pageKey`。
6. 将新页面放到管理员指定位置，未指定时放入“未分类功能”。

系统升级和管理员调整菜单时，不得粗暴清空终端用户个人配置。

### 8.7 安全和恢复规则

- 菜单隐藏不等于取消权限，后端接口必须继续独立校验权限。
- 无权页面即使存在旧的本地配置引用，也必须在权限过滤阶段被移除。
- 删除一级目录只能解除菜单归属，不能删除真实页面或业务数据。
- 必须保留一个不可被普通配置隐藏的菜单管理/恢复入口。
- 必须支持“恢复系统默认”和“恢复管理员默认”两个层级。
- 管理员基础配置变更必须写审计日志。

### 8.8 默认菜单模板

系统可提供多套初始模板，模板只用于初始化或管理员主动恢复：

- 按业务流程。
- 按人、机、料、法、环。
- 按部门职责。
- 简洁生产版。
- 管理员自定义。

用户选择模板后可继续改名、分组和排序，不要求从空白菜单开始。

### 8.9 新功能接入检查

1. 是否注册了唯一且稳定的 `pageKey`。
2. 是否声明了默认名称、权限资源、图标和支持终端。
3. 是否在管理员未分类时安全进入“未分类功能”。
4. 是否经过权限过滤后才应用个人布局。
5. 是否保持现有管理员配置和用户个人配置。
6. 是否同时验证桌面端、移动端和恢复默认路径。
