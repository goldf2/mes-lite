# MES-lite 响应式断点与验收矩阵

本文定义主流分辨率下的布局规划和实际模拟验收流程。MES-lite 不只写 CSS 响应式规则，关键页面必须用浏览器实际模拟多个尺寸，确认页面真实呈现效果后再交付。

## 1. 基本判断

按多个主流分辨率模拟页面效果是正确的开发流程，尤其适合 MES/ERP 这类信息密度高、操作多的系统。原因：

- 断点规则只能推测布局，不能发现文字挤压、按钮重叠、弹窗超出、表格撑破页面等真实问题。
- 桌面端和移动端的目标不同，大屏追求效率，手机追求轻量和可触控。
- 同一页面在 1024、1366、1920 宽度下的工具栏、表格、详情面板表现会明显不同。
- 未来接入小程序时，提前建立移动端验收矩阵可以减少重复返工。

## 2. 主流分辨率矩阵

| 类型 | 分辨率 | 主要用途 | 布局策略 |
| --- | --- | --- | --- |
| 小手机 | 360 x 800 | 低端安卓、小屏手机 | 单列卡片、底部导航、操作菜单 |
| 标准手机 | 390 x 844 | iPhone 常见尺寸 | 单列卡片、关键词 + 筛选、底部抽屉 |
| 大手机 | 414 x 896 | 大屏手机 | 单列或紧凑双区块，操作仍进菜单 |
| 平板竖屏 | 768 x 1024 | iPad 竖屏、折叠屏 | 卡片优先，可显示更多筛选摘要 |
| 小桌面 | 1024 x 768 | 小笔记本、平板横屏 | 左侧导航压缩，顶部工具栏允许换行 |
| 常规笔记本 | 1366 x 768 | 常见办公电脑 | 表格优先，主操作首层显示 |
| 中等桌面 | 1440 x 900 | 常见办公显示器 | 表格 + 详情弹窗，工具栏一行优先 |
| 高分屏笔记本 | 1536 x 864 | Mac/Windows 缩放后常见 | 表格信息密度提升，卡片多列 |
| 标准大屏 | 1920 x 1080 | 办公大屏 | 表格 + 分栏详情，可展示更多字段 |
| 超宽/2K | 2560 x 1440 | 大显示器 | 限制单行阅读宽度，允许左右分栏 |

## 3. 布局断点规划

### 3.1 小于 640px

- 手机布局。
- 单列卡片为主。
- 表格默认不作为主视图。
- 顶部只保留标题、关键词、筛选、操作菜单。
- 详情、编辑、上传、文件查看全部全屏或底部抽屉。

### 3.2 640px 到 767px

- 大手机/小平板布局。
- 可以增加少量摘要字段。
- 仍不使用完整桌面表格。
- 操作按钮仍以弹出菜单为主。

### 3.3 768px 到 1023px

- 平板布局。
- 可以显示更多卡片列或紧凑列表。
- 左侧导航可折叠为图标或使用底部/侧边抽屉。
- 详情可使用全屏或宽弹窗。

### 3.4 1024px 到 1279px

- 小桌面布局。
- 左侧导航显示，但宽度可以压缩。
- 顶部工具栏允许换行。
- 表格作为主视图，但列数量要克制。
- 行内操作保持最右列。

### 3.5 1280px 到 1535px

- 标准桌面布局。
- 顶部工具栏优先一行显示。
- 表格显示完整核心字段。
- 详情使用大弹窗。
- 卡片视图最多 2 到 4 列。

### 3.6 1536px 到 1919px

- 宽桌面布局。
- 可以增加分栏、更多统计卡、更多表格字段。
- 详情弹窗可以更宽，但表单仍保持 2 到 3 列。
- 不因为宽度大就无限拉长文本行。

### 3.7 大于等于 1920px

- 大屏布局。
- 允许表格和详情并列。
- 统计页允许多图表并列。
- 内容区应保持可读宽度，避免文字行过长。
- 可以展示更多辅助字段，但不能破坏核心字段扫描效率。

## 4. 每个尺寸必须检查的问题

### 4.1 通用检查

- 页面主体是否出现横向滚动。
- 顶部工具栏是否按钮重叠。
- 左侧导航或底部导航是否遮挡内容。
- 筛选弹窗是否超出屏幕。
- 操作菜单是否能点击空白关闭。
- 详情弹窗是否能完整关闭。
- 表格是否只在自身容器内横向滚动。
- 按钮文字是否溢出。
- 状态标签是否挤压主字段。
- 文件查看器是否能正常显示图片和 PDF。

### 4.2 桌面端检查

- 主要操作是否在首层显示。
- 表格列顺序是否稳定。
- 行内操作是否在最右侧。
- 批量扫描时是否需要频繁打开弹窗。
- 1024px 下工具栏换行后是否仍可读。
- 1920px 下内容是否过度拉伸。

### 4.3 移动端检查

- 是否默认使用卡片。
- 每张卡片是否只显示核心字段。
- 操作是否收敛到菜单或底部抽屉。
- 筛选是否使用底部抽屉或全屏页。
- 上传/拍照入口是否足够大。
- 触控区域是否不小于 36px。

## 5. 模拟验收流程

每次修改页面布局、工具栏、列表、卡片、弹窗、文件查看器时，必须执行以下流程：

1. 启动本地服务。
2. 使用浏览器或 Playwright 设置目标 viewport。
3. 分别打开相关业务页面。
4. 对每个主流尺寸检查布局、按钮、筛选、弹窗、详情。
5. 对发现的问题回到组件或页面修复。
6. 再次模拟受影响尺寸。
7. 最后执行类型检查和构建。

建议保留的检查命令：

```bash
npx tsc --noEmit --pretty false
npm run build
```

后续可以增加：

```bash
npm run verify:responsive
```

该脚本应自动打开关键页面，在多个 viewport 下检查页面级横向滚动、按钮重叠、工具栏可见性和主要弹窗可用性。

## 7. 交付标准

- 桌面端至少验证：1024 x 768、1366 x 768、1440 x 900、1920 x 1080。
- 移动端至少验证：360 x 800、390 x 844、414 x 896、768 x 1024。
- 页面级 `document.documentElement.scrollWidth` 不得大于 `clientWidth`。
- 顶部工具栏、底部导航、操作菜单、详情弹窗不得互相遮挡。
- 主要业务操作在当前设备形态下必须可发现。
- 验证结果应记录在测试说明或修复报告中，不能只凭主观印象。
