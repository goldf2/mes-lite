# MES-lite 界面开发规则

本文用于后续界面开发前自查，目标是让所有页面的视觉、按钮、筛选、表格、卡片、弹窗保持统一。页面不应各自自由拼样式，通用视觉必须集中在公共组件和设计令牌中维护。

本文件只定义共通规则。桌面端和移动端必须分开设计、分开验收：

- 桌面端规则见：`docs/minierp/桌面端界面开发规范.md`
- 移动端规则见：`docs/minierp/移动端界面开发规范.md`
- 响应式断点和模拟验收见：`docs/minierp/响应式断点与验收矩阵.md`
- 草图到实现的协作流程见：`docs/minierp/界面草图到实现流程.md`
- 吸收低代码平台优点的内部配置化框架见：`docs/minierp/配置化页面框架规划.md`

## 1. 核心原则

- 视觉规则集中：按钮、状态标签、表格、卡片、筛选、弹窗、文件查看器必须由公共组件提供。
- 业务页面只负责数据、权限和业务动作，不直接决定按钮颜色、行高、圆角、阴影、状态颜色。
- 一个功能只允许一个标准文案，例如同类新增按钮统一写“新增”，不要在页面里同时出现“新建”“新增来料”“创建”等混用。
- 一个功能只允许一套标准尺寸，例如行内“详情”按钮不能有的高 28px、有的高 36px。
- 一个功能尽量固定一个位置，例如列表行操作统一在最右列，页面主操作统一在顶部工具栏右侧。
- 不通过复制粘贴页面代码保持一致；通过 `Button`、`DataTable`、`ResourceToolbar`、`DetailPanel` 等组件保持一致。

## 2. 推荐组件分层

后续新增或重构页面时，优先沉淀到以下目录：

```txt
app/components/ui/
  Button.tsx
  StatusBadge.tsx
  DataTable.tsx
  EntityCard.tsx
  FilterPopover.tsx
  Toolbar.tsx
  Modal.tsx
  FormGrid.tsx
  FileViewer.tsx

app/components/page/
  ResourcePage.tsx
  ResourceToolbar.tsx
  ResourceList.tsx
  DetailPanel.tsx

app/styles/
  design-tokens.ts
  status-styles.ts
```

未抽出公共组件前，也必须按本文规则写页面；抽出公共组件后，禁止业务页面继续手写同类样式。

## 3. 按钮规范

### 3.1 标准按钮文案

- 新增：统一写“新增”。页面标题已经说明模块时，不写“新增来料”“新增派工单”“新增指导书”。
- 详情：统一写“详情”。用于打开资料、单据、指导书等详情面板。
- 全屏打开：统一写“全屏打开”。用于直接查看图片、PDF、指导书、图纸等展示内容。
- 编辑：统一写“编辑”。详情内编辑基础信息时可写“编辑信息”。
- 保存：统一写“保存”。保存后进入上传流程时可写“保存并上传文件”。
- 取消：统一写“取消”。
- 关闭：统一写“关闭”或使用关闭图标，但同一弹窗内不能同时混用多个关闭文案。
- 归档：统一写“归档”。所有软删除都用归档文案。
- 恢复：统一写“恢复”或“恢复归档”，归档记录页面优先写“恢复归档”。
- 导入、导出、上传、下载、打印、调整：必须使用这些固定词，不再另写近义词。

### 3.2 按钮视觉级别

- `primary`：页面主确认动作，例如保存、确认、全屏打开。
- `create`：新增类动作，例如新增、新建首条记录。
- `secondary`：普通动作，例如详情、导出、下载、取消。
- `archive`：归档动作，使用琥珀色边框或弱警示样式。
- `danger`：不可逆或高风险动作，使用红色；当前系统原则上少用。
- `ghost/icon`：顶部菜单、关闭、视图切换等弱操作。

### 3.3 按钮尺寸

- 顶部工具栏按钮：高度 36px 到 40px，文字 12px 到 14px。
- 表格行内按钮：高度 28px 到 32px，文字 12px。
- 卡片底部按钮：高度 30px 到 34px，文字 12px。
- 弹窗底部按钮：高度 36px 到 40px，文字 14px。
- 图标按钮：宽高一致，常规 32px；移动端不得小于 36px 可点击区域。
- 同一行内按钮必须同高、同圆角、同字号。

### 3.4 按钮位置

- 页面主操作放顶部工具栏右侧：新增、导入、导出、调整。
- 行内操作放最右列，顺序固定：全屏打开 / 详情 / 编辑 / 上传 / 打印 / 归档。
- 卡片操作放卡片底部右侧，顺序同表格行内操作。
- 弹窗底部操作放右侧，顺序固定：取消、保存。
- 关闭按钮放弹窗右上角。
- 不能把高频操作只藏在二级菜单里；二级菜单只放低频、危险或批量操作。

## 4. 顶部工具栏

- 顶部首层固定保留：关键词搜索、统一“筛选”按钮、已选筛选摘要。
- 类别、状态、客户、供应商、物料、文件类型、排序、字段显示等筛选对象统一放入点击弹出的筛选窗口。
- 筛选按钮需要能表达当前是否有筛选条件，优先显示数量，例如“筛选 3”。
- 新增、导入、导出、上传、调整等高频操作按钮必须暴露在首层。
- 顶部工具栏不得产生页面级横向滚动条；窄屏时优先换行、压缩宽度，筛选窗口可作为手机底部抽屉或浮层。
- 仪表盘、统计分析等总览页不默认提供卡片/列表切换，避免变成生硬表格。

## 5. 筛选控件

- 勾选类筛选在筛选窗口内使用可见复选框。
- 不在显示界面提供筛选项手动排序、拖动排序、左移右移按钮，避免浪费空间。
- 筛选状态应尽量保存在浏览器本地，避免刷新后丢失用户习惯。
- 物料、客户、供应商等数据量可能较大的选项，优先使用“输入搜索后选择”，避免长下拉框。
- 搜索选择不能只依赖前端已加载的少量数据；输入关键字时应调用后端查询。
- 客户筛选必须和页面显示逻辑一致，例如作业指导书绑定物料时，应优先按物料客户判断有效客户。

## 6. 表格规范

- 所有表格优先复用 `DataTable`。
- 表头背景、字号、行高、边框、空状态、分页位置必须统一。
- 表格可以在自身容器内横向滚动，但不能撑破页面主容器。
- 操作列固定在最右侧；字段显示过多时优先做字段控制，不随意缩小所有列。
- ID、编码、单号使用等宽字体；名称、备注允许换行或截断。
- 金额、数量、库存应有统一格式化方法，不能每个页面自行 `toFixed`。

## 7. 卡片规范

- 卡片只用于移动端优先视图、资料卡、实体摘要，不用于包裹整个页面大区块。
- 卡片圆角不超过 8px，避免过度装饰。
- 卡片信息顺序固定：编码/状态、名称、关键属性、数量/金额、操作。
- 卡片操作放底部右侧；不要把主要按钮放在卡片中部打断阅读。
- 卡片和列表必须展示同一批核心字段，不能卡片有字段、列表缺字段导致理解不一致。

## 8. 详情、编辑、弹窗

- 详情页优先使用 `DetailPanel` 或统一 `Modal`。
- 详情顶部固定显示：编码、名称/标题、状态、关闭按钮。
- 详情内结构固定为：基础信息、业务信息、附件/原始凭证、操作记录。
- 编辑态在详情内切换，不单独弹出另一个样式完全不同的编辑窗口。
- 新增弹窗使用 `FormGrid`，字段按 2 到 3 列布局，避免一长条表单。
- 上传入口必须在详情里清晰可见，支持拖放时要有明确的拖放区域。

## 9. 文件与图片查看

- 图片、PDF、作业指导书、单据凭证统一使用 `FileViewer`。
- 列表首层常用展示动作统一用“全屏打开”。
- 详情内附件可使用“打开”，但其行为也应进入同一个 `FileViewer`。
- 图片查看应支持放大、缩小、旋转、复位。
- PDF 查看应支持浏览器内置多页滚动。
- 文件归档只影响记录显示，不物理删除原始文件，除非后续明确增加物理清理功能。

## 10. 状态标签

- 状态颜色集中在 `status-styles.ts`。
- 同一状态在所有页面必须同色同文案。
- 常用状态建议：
  - 草稿：灰色。
  - 待处理、待审核、待收货：琥珀色。
  - 进行中、已派工：蓝色。
  - 已完成、已收货、已发货：绿色。
  - 拒绝、异常、库存不足：红色。
  - 已归档、已取消：灰色弱化。

## 11. 平台分离规则

- 桌面端服务大屏高效率录入、扫描、比对、批量处理，应优先展示表格、工具栏、筛选摘要、行内操作。
- 移动端服务窄屏快速查看、拍照上传、现场确认，应优先使用卡片、弹出菜单、底部抽屉、全屏详情。
- 不能用同一套布局硬适配两端；只能共用组件语义、按钮文案、状态颜色、数据格式。
- 同一功能在桌面端可以暴露多个高频按钮；在移动端应收敛到“更多”菜单或底部操作面板。
- 桌面端和移动端修改都必须分别检查，不能只看一个视口。
- 关键页面必须按主流分辨率矩阵做实际模拟，不能只凭 CSS 断点推测效果。

## 12. 权限、归档与危险操作

- 没有权限的操作应隐藏或置灰，不能显示后点击才报错。
- “删除”默认是归档/软删除，界面文案统一写“归档”。
- 高风险动作必须二次确认。
- 归档记录必须能在系统管理中恢复。
- 所有会改变数据的操作必须写入审计日志。

## 13. 禁止事项

- 禁止在业务页面里自由写一套新的按钮颜色、表格行高、卡片样式。
- 禁止同一功能出现多个文案，例如“打开/查看/详情”混用。
- 禁止同一页面顶部同时出现两排筛选按钮。
- 禁止把新增、导入、导出等高频操作藏进二级菜单。
- 禁止页面级横向滚动。
- 禁止卡片嵌套卡片。
- 禁止为了临时显示数据把表格列压到不可读。
- 禁止新增物理删除入口。

## 14. 菜单和页面注册规则

- 新增功能页必须先进入统一页面注册表，不得在桌面侧栏、移动端底部栏和“更多”面板里分别手写三份菜单。
- 页面 `pageKey`、路由、权限资源与显示名称必须分离；改名和移动菜单不得改变权限或业务行为。
- 系统管理员负责一级目录、二级名称、默认顺序、启停和可隐藏范围。
- 终端用户只能在管理员配置和自身权限内调整个人顺序、折叠、隐藏和移动端常用入口。
- 第一阶段个人布局保存到带版本号的 `localStorage`；后续可同步到用户账户，并提供 JSON 导入导出。
- 菜单必须先应用管理员配置和权限过滤，再应用个人布局。
- 新页面不得覆盖现有菜单配置；未分类时放入“未分类功能”。
- 必须保留不可被普通用户隐藏的菜单管理和恢复默认入口。
- 完整数据模型、合并算法、安全边界和默认模板见 `docs/minierp/配置化页面框架规划.md`。

## 15. 新模块开发前检查

1. 是否使用统一按钮文案、尺寸、位置。
2. 是否有首层关键词筛选。
3. 是否有统一筛选按钮和筛选摘要。
4. 是否把高频操作按钮放在首层。
5. 是否同时符合桌面端和移动端规范。
6. 是否避免页面级横向滚动。
7. 是否支持卡片 / 列表视图或明确说明为什么不需要。
8. 是否使用统一状态标签和统一状态文案。
9. 是否使用统一详情弹窗或详情面板。
10. 是否支持软删除 / 归档与恢复路径。
11. 是否按响应式验收矩阵检查主流分辨率。
12. 是否完成 `npx tsc --noEmit --pretty false` 和 `npm run build` 验证。
13. 是否通过统一页面注册表接入，并保持管理员与终端用户的已有菜单配置。

## 16. 后续重构顺序

1. 先抽 `Button`、`StatusBadge`、`FilterPopover`、`FileViewer`。
2. 再抽 `ResourceToolbar`、`DataTable`、`EntityCard`。
3. 先迁移作业指导书、物料管理，因为这两个页面最能验证图片、筛选、详情和列表规范。
4. 再迁移来料、库存、派工、发货、退货。
5. 最后整理系统管理、权限管理、统计分析。
