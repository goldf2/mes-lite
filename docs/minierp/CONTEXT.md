# MiniERP 项目上下文

## 项目定位

MiniERP 是一个轻量企业管理系统，第一阶段服务小商贸、小门店、小仓库或小型制造团队的日常经营管理。

系统优先解决内部管理问题，不把第一版做成面向消费者的商城。微信小程序接入先定位为移动端管理入口，主要给老板、员工、仓管、采购、销售使用。

## 第一版成功标准

- 能管理一个或多个客户公司，也就是多租户。
- 每个公司可以维护员工、角色、商品、客户、供应商、仓库。
- 能记录采购入库、销售出库、库存调整和库存盘点。
- 能在小程序端完成常见移动操作：查库存、开销售单、收货入库、盘点、看经营摘要。
- 业务数据与小程序主体解耦，后续可以从开发商主体迁移到客户主体。

## 暂不包含

- C 端商城、消费者下单、会员营销。
- 微信支付收款闭环。
- 发票、税务、财务凭证自动化。
- 复杂生产排程、MRP 运算、工序成本核算。
- 多平台电商订单聚合。

## 主体与小程序策略

第一版小程序可以注册在开发商主体下，定位为企业管理工具或企业办公管理工具。

为了后续可迁移：

- 租户数据必须归属于客户公司，而不是归属于小程序主体。
- 小程序 AppID、微信支付商户号、服务器域名、客户数据归属要分层记录。
- 不在开发商主体小程序里承载客户公司的对外经营行为。
- 若客户后续需要独立持有小程序，可通过微信小程序主体变更或独立部署迁移。

## 关键术语

- 租户：一个使用 MiniERP 的客户公司或经营主体。
- 门店：租户下的经营点，可选；第一版可以只有一个默认门店。
- 仓库：库存归属地点，可以是实体仓库、门店仓、车仓。
- SKU：实际库存管理单位。
- 商品：业务展示和销售层面的货品，可以对应一个或多个 SKU。
- 单据：采购、销售、调拨、盘点、调整等业务动作的记录。
- 库存流水：任何导致库存数量变化的明细记录。
- 小程序主体：微信小程序账号的法律主体，可能是开发商，也可能是客户公司。
- 业务主体：实际经营、履约、持有客户经营数据的公司。
