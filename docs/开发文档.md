# MES-lite 工厂生产系统开发文档

## 1. 项目概述

### 1.1 项目定位

MES-lite 是一个轻量级制造执行系统（Manufacturing Execution System），旨在实现工厂生产全流程的数字化管理，包括：

- 生产工单管理
- 物料领用与库存控制
- 工序报工与进度追踪
- 质检记录与成品入库
- 质量数据记录

### 1.2 技术栈

| 层级 | 技术 | 版本 | 说明 |
|------|------|------|------|
| 框架 | Next.js | 14.2 | App Router，全栈框架 |
| ORM | Prisma | 5.22 | 数据库操作 |
| 数据库 | SQLite / PostgreSQL | — | 开发用 SQLite，生产用 PostgreSQL |
| 验证 | Zod | 3.23 | API 参数校验 |
| 样式 | Tailwind CSS | 3.4 | 页面样式 |

## 2. 数据库模型设计

### 2.1 实体关系图

```
Product ──── 1:1 ──── BOM ──── 1:N ──── BOMItem ──── N:1 ──── Material
  │                        │
  │                        └─── 1:N ──── ProcessRoute ──── 1:N ──── ProcessStep
  │
  └─── 1:N ──── ProductionOrder ──── 1:N ──── PickItem
                      │
                      ├─── 1:N ──── WorkReport
                      ├─── 1:N ──── QCRecord
                      └─── 1:N ──── StockIn

Material ──── 1:1 ──── Stock ──── 1:N ──── StockLog
Product  ──── 1:1 ──── Stock
```

### 2.2 核心模型详解

#### 2.2.1 Product（产品）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键，CUID |
| sku | String | 产品编码，唯一 |
| name | String | 产品名称 |
| category | String | 分类（轴承/齿轮/密封件） |
| unit | String | 单位，默认"件" |
| description | String? | 描述 |

#### 2.2.2 Material（原材料）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| code | String | 物料编码，唯一 |
| name | String | 物料名称 |
| spec | String? | 规格型号 |
| unit | String | 单位 |

#### 2.2.3 BOM（物料清单）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| productId | String | 关联产品，唯一 |
| version | String | 版本号，默认"v1" |
| isActive | Boolean | 是否启用 |

#### 2.2.4 BOMItem（BOM 子项）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| bomId | String | 关联 BOM |
| materialId | String | 关联物料 |
| quantity | Float | 用量（不含损耗） |
| unit | String | 单位 |
| wastageRate | Float | 损耗率，默认 0 |

#### 2.2.5 ProcessRoute（工艺路线）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| productId | String | 关联产品 |
| name | String | 路线名称 |
| isDefault | Boolean | 是否默认路线 |

#### 2.2.6 ProcessStep（工序步骤）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| routeId | String | 关联工艺路线 |
| stepNo | Int | 工序序号 |
| name | String | 工序名称 |
| defaultTime | Int? | 标准工时（分钟） |
| workstation | String? | 工作中心 |

#### 2.2.7 ProductionOrder（生产工单）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| orderNo | String | 工单号，唯一（WO-YYYYMMDD-XXX） |
| productId | String | 关联产品 |
| planQty | Int | 计划产量 |
| completeQty | Int | 完成数量 |
| scrapQty | Int | 报废数量 |
| status | String | 状态 |

**工单状态流转：**

```
DRAFT → CONFIRMED → PICKED → RUNNING → QC_WAITING → QC_DONE → COMPLETED
     ↘ CANCELLED（任意非已入库状态可取消）
```

#### 2.2.8 PickItem（领料项）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| orderId | String | 关联工单 |
| materialId | String | 关联物料 |
| requiredQty | Float | 需求数量（含损耗） |
| actualQty | Float | 实际领用数量 |
| returnedQty | Float | 退回数量 |
| status | String | PENDING / PARTIAL / COMPLETED / RETURNED |

#### 2.2.9 WorkReport（工序报工）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| orderId | String | 关联工单 |
| stepId | String | 关联工序 |
| workerName | String | 工人姓名 |
| startTime | DateTime | 开始时间 |
| endTime | DateTime? | 结束时间 |
| goodQty | Int | 合格品数量 |
| badQty | Int | 不良品数量 |
| badReason | String? | 不良原因 |

#### 2.2.10 Stock（库存）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| materialId | String? | 关联原材料（二选一） |
| productId | String? | 关联产品（二选一） |
| qty | Float | 当前库存 |
| reservedQty | Float | 已预留数量 |
| availableQty | Float | 可用数量 |

#### 2.2.11 StockLog（库存日志）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键 |
| stockId | String | 关联库存 |
| type | String | IN / OUT / PICK / RETURN / STOCK_IN / ADJUST |
| qty | Float | 变动数量 |
| beforeQty | Float | 变动前数量 |
| afterQty | Float | 变动后数量 |
| refType | String? | 关联单据类型 |
| refId | String? | 关联单据 ID |

## 3. API 接口详细说明

### 3.1 生产工单接口

#### 3.1.1 POST /api/orders — 创建工单

**请求体：**

```json
{
  "productId": "string",      // 产品 ID，必填
  "planQty": 100,             // 计划产量，必填，正整数
  "note": "string"            // 备注，可选
}
```

**响应（201）：**

```json
{
  "data": {
    "id": "string",
    "orderNo": "WO-20260702-001",
    "productId": "string",
    "planQty": 100,
    "status": "DRAFT",
    "createdAt": "2026-07-02T10:00:00.000Z"
  }
}
```

**业务逻辑：**
1. 检查产品是否存在且有 BOM 和工艺路线
2. 生成工单号（WO-YYYYMMDD-XXX）
3. 创建工单记录
4. 根据 BOM 计算物料需求（用量 × 数量 × (1+损耗率)）
5. 预留库存（减少 availableQty，增加 reservedQty）

#### 3.1.2 GET /api/orders — 工单列表

**查询参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| status | String | 状态筛选（可选） |
| page | Int | 页码，默认 1 |
| pageSize | Int | 每页数量，默认 20 |

**响应（200）：**

```json
{
  "data": [
    {
      "id": "string",
      "orderNo": "WO-20260702-001",
      "status": "RUNNING",
      "planQty": 100,
      "completeQty": 50,
      "product": {
        "id": "string",
        "name": "深沟球轴承 6204",
        "sku": "ZC-6204"
      },
      "_count": {
        "reports": 3,
        "picks": 2
      }
    }
  ],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "total": 50,
    "totalPages": 3
  }
}
```

#### 3.1.3 GET /api/orders/:id — 工单详情

**响应（200）：**

```json
{
  "data": {
    "id": "string",
    "orderNo": "WO-20260702-001",
    "status": "RUNNING",
    "product": { ... },
    "picks": [{ ... }],
    "reports": [{ ... }],
    "qcRecords": [{ ... }],
    "stockIns": [{ ... }],
    "currentStepId": "string",    // 当前应报工的工序 ID
    "routeSteps": [{ ... }]       // 所有工序步骤
  }
}
```

#### 3.1.4 POST /api/orders/:id/pick — 领料

**请求体：**

```json
{
  "items": [
    {
      "pickItemId": "string",    // 领料项 ID，必填
      "actualQty": 35.5,         // 实际领用数量，必填
      "pickedBy": "张三"         // 领料人，必填
    }
  ]
}
```

**响应（200）：**

```json
{
  "success": true,
  "message": "领料完成"
}
```

**业务逻辑：**
1. 校验工单状态必须为 CONFIRMED
2. 校验库存充足
3. 扣减库存（qty 和 availableQty）
4. 记录库存日志（PICK 类型）
5. 更新领料项状态为 COMPLETED
6. 若所有领料完成，工单状态转为 PICKED

#### 3.1.5 POST /api/orders/:id/reports — 工序报工

**请求体：**

```json
{
  "stepId": "string",           // 工序 ID，必填
  "workerName": "张三",         // 工人姓名，必填
  "workerId": "string",         // 工号，可选
  "goodQty": 98,                // 合格品数量，必填
  "badQty": 2,                  // 不良品数量，必填
  "badReason": "尺寸偏差",      // 不良原因，可选
  "remark": "string",           // 备注，可选
  "photoUrls": ["url1"]         // 照片 URL，可选
}
```

**响应（200）：**

```json
{
  "success": true,
  "message": "报工成功"
}
```

**业务逻辑（防呆校验）：**
1. 校验工单状态必须为 PICKED 或 RUNNING
2. 校验工序属于该产品的工艺路线
3. **防呆**：检查上一工序是否已完成（endTime 不为空）
4. 若该工序已有未结束的报工记录，更新之；否则新建
5. 更新工单状态：
   - 若所有工序完成 → QC_WAITING
   - 若首次报工 → RUNNING

#### 3.1.6 POST /api/orders/:id/stock-in — 成品入库

**请求体：**

```json
{
  "qty": 95,                    // 入库数量，必填
  "batchNo": "B20260702001",    // 批次号，可选
  "inBy": "李四",               // 入库人，必填
  "note": "string"              // 备注，可选
}
```

**响应（200）：**

```json
{
  "success": true,
  "message": "入库成功"
}
```

**业务逻辑：**
1. 校验工单状态必须为 QC_DONE
2. 创建入库记录
3. 更新成品库存
4. 记录库存日志（STOCK_IN 类型）
5. 更新工单状态为 COMPLETED

#### 3.1.7 PATCH /api/orders/:id/cancel — 取消工单

**请求体：**

```json
{
  "reason": "客户取消订单"       // 取消原因，必填
}
```

**响应（200）：**

```json
{
  "success": true,
  "message": "工单 WO-20260702-001 已取消，物料已退库"
}
```

**业务逻辑（事务回退）：**
1. 禁止取消：已完成（COMPLETED）或已有入库记录的工单
2. 禁止重复取消
3. 事务回退：
   - 已领物料回退（增加 qty 和 availableQty，减少 reservedQty）
   - 记录退料日志（RETURN 类型）
   - 标记领料项为 RETURNED
   - 报工记录备注标记"工单取消作废"
4. 更新工单状态为 CANCELLED

## 4. 业务流程说明

### 4.1 完整生产流程

```
1. 创建工单（DRAFT）
   ↓
2. 确认工单（CONFIRMED）—— 当前需手动修改数据库
   ↓
3. 领料（PICKED）
   ↓
4. 工序报工（RUNNING → QC_WAITING）
   ↓
5. 质检（QC_DONE）—— 当前需手动修改数据库
   ↓
6. 成品入库（COMPLETED）
```

### 4.2 库存变动规则

| 操作 | qty | reservedQty | availableQty |
|------|-----|-------------|--------------|
| 创建工单（预留） | 不变 | +requiredQty | -requiredQty |
| 领料 | -actualQty | 不变 | -actualQty |
| 取消工单（退料） | +actualQty | -requiredQty | +actualQty |
| 成品入库 | +qty | 不变 | +qty |
| 盘点调整 | → newQty | 不变 | newQty - reservedQty |

### 4.3 防呆机制

1. **工序顺序防呆**：报工必须上一工序已完成
2. **状态流转防呆**：工单必须按顺序流转，非法跳转返回 400
3. **库存校验防呆**：领料时校验库存充足
4. **取消限制防呆**：已入库工单不可取消

## 5. 测试用例

### 5.1 测试步骤

#### 步骤 1：获取产品 ID

```bash
curl http://localhost:3000/api/orders
# 返回空列表，但可以通过 prisma studio 查看产品
npx prisma studio
```

#### 步骤 2：创建工单

```bash
curl -X POST http://localhost:3000/api/orders \
  -H "Content-Type: application/json" \
  -d '{"productId":"<产品ID>","planQty":100}'
```

#### 步骤 3：确认工单（手动）

打开 Prisma Studio，将工单 status 改为 CONFIRMED。

#### 步骤 4：查看工单详情

```bash
curl http://localhost:3000/api/orders/<工单ID>
# 查看 picks 和 currentStepId
```

#### 步骤 5：领料

```bash
curl -X POST http://localhost:3000/api/orders/<工单ID>/pick \
  -H "Content-Type: application/json" \
  -d '{"items":[{"pickItemId":"<领料项ID>","actualQty":35,"pickedBy":"张三"}]}'
```

#### 步骤 6：工序报工

```bash
curl -X POST http://localhost:3000/api/orders/<工单ID>/reports \
  -H "Content-Type: application/json" \
  -d '{"stepId":"<工序ID>","workerName":"张三","goodQty":98,"badQty":2}'
```

#### 步骤 7：查看库存

```bash
curl http://localhost:3000/api/stocks
```

#### 步骤 8：取消工单（测试回退）

```bash
curl -X PATCH http://localhost:3000/api/orders/<工单ID>/cancel \
  -H "Content-Type: application/json" \
  -d '{"reason":"客户取消订单"}'
```

### 5.2 边界测试

| 测试场景 | 预期结果 |
|----------|----------|
| 上一工序未完成时报工 | 返回 400 错误 |
| 库存不足时领料 | 返回 400 错误 |
| 已完成工单取消 | 返回 400 错误 |
| 重复取消工单 | 返回 400 错误 |

## 6. 部署指南

### 6.1 开发环境

```bash
npm install
npx prisma generate
npx prisma migrate dev --name init
npx prisma db seed
npm run dev
```

### 6.2 生产环境

#### 使用 PostgreSQL

修改 `prisma/schema.prisma`：

```prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

修改 `.env`：

```
DATABASE_URL="postgresql://user:password@host:5432/database?schema=public"
```

#### 部署到 Vercel

```bash
vercel --prod
```

**环境变量配置：**
- DATABASE_URL：数据库连接字符串

## 7. 代码规范

### 7.1 命名规范

- 文件：小写+短横线（`route.ts`, `seed.ts`）
- 变量：驼峰式（`orderId`, `planQty`）
- 模型：帕斯卡式（`ProductionOrder`, `WorkReport`）
- 路由参数：`:id`, `:orderId`

### 7.2 API 响应格式

```json
{
  "data": { ... },           // 成功数据
  "pagination": { ... },     // 分页信息（可选）
  "success": true,           // 操作结果
  "message": "提示信息"       // 提示信息
}
```

错误响应：

```json
{
  "error": "错误描述",
  "details": [ ... ]         // 详细错误（可选）
}
```

### 7.3 事务使用

涉及多表操作的业务必须使用 Prisma 事务：

```typescript
await prisma.$transaction(async (tx) => {
  // 操作 1
  // 操作 2
  // ...
})
```
